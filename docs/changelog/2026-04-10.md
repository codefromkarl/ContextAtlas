# ContextAtlas 更新总结（2026-04-10）

本轮更新不是单一功能点，而是一次完整的日内收口：

> 先把检索运行时、并发隔离、资源清理和错误边界收稳，再把代码图谱能力正式接进 ContextAtlas，并让图谱第一次进入默认检索主路径。

## 一、为什么要更新

这次更新的背景，来自同一天内两类问题的连续收口。

### 1. 检索与运行时链路先暴露出了稳定性问题

4 月 10 日凌晨的几笔提交，集中解决的是 retrieval / search runtime 的基础问题：

- cached 资源关闭路径分散，scanner / indexer / vector store 的清理边界不够统一
- MCP 工具在检索失败、JSON-RPC 失败或 `fetch failed` 场景下，错误语义不够清晰
- embedding 配置和全局限流锁在真实并发环境里还需要进一步收紧
- query-aware search config 在并发请求下存在隔离风险
- search runtime wiring 还没有完全拆出清晰边界

这些问题如果不先处理，后续把图谱能力接进主路径，只会把复杂度叠加到一条还不够稳的链路上。

### 2. 代码图谱能力在同一天从“设计/底层能力”推进到了“主路径能力”

在运行时基础问题收口之后，4 月 10 日白天的两笔关键提交把另一条线推到位了：

- `32c6653 feat: add code graph analysis pipeline`
- `cdf1d42 feat: enable graph context by default`

这意味着当天真正发生的，不只是“默认值从 false 改成 true”，而是：

- ContextAtlas 内部第一次落地了完整代码图谱管线
- 紧接着又把这项能力从独立图工具推进到了默认检索出口

### 3. 对比 GitNexus 后，判断重点不再是“要不要有图工具”，而是“图信号要不要进入默认检索主路径”

对比 GitNexus 这类图谱型代码检索产品后，差异已经很明确：

- 图谱价值不只在独立 `context` / `impact` 工具
- 更在于默认检索结果是否天然带有结构化关系信息
- 工具是否能一次返回足够多的上下文，而不是让 agent 通过多轮补查自己拼图

因此 4 月 10 日真正要解决的问题，不是“补一个 graph tool”，而是：

- 先把 ContextAtlas 的运行时链路收稳
- 再让刚落地的图谱能力进入最常用的 `codebase-retrieval` 主路径

## 三、如何更新

### 1. 先收稳运行时与资源边界

这部分主要由 `7755018`、`047917e`、`4d3f100`、`57e0ccd`、`75ebf30` 完成。

重点包括：

- 把 cached resource cleanup 集中到统一关闭路径
- 让 scanner / indexer / vector store 的生命周期边界更清晰
- 强化 MCP 检索错误处理，尤其是 retrieval stage 的失败标注
- 收紧 embedding 配置和全局限流锁逻辑
- 隔离 query-aware search config，避免并发请求互相污染
- 拆出 search runtime wiring、pipeline callbacks 和 runtime provider 边界

这一步的意义是：把检索主链路从“能跑”推进到“在并发、失败和资源回收场景下更稳”。

### 2. 正式落地代码图谱分析管线

这部分由 `32c6653` 完成，是当天最大的能力增量。

当天落地的核心内容包括：

- `GraphStore`
- `SymbolExtractor`
- `ChangeDetector`
- `ExecutionTracer`
- `graph_context`
- `graph_impact`
- `graph_query`
- `detect_changes`
- scanner graph 持久化
- `GraphExpander` 的 graph-first / fallback-aware 扩展

这一步解决的是：ContextAtlas 不再只有“chunk 级语义检索 + 文件级扩展”，而开始具备符号级关系查询和图谱分析能力。

### 3. 把图谱能力接进默认检索主路径

这部分由 `cdf1d42` 完成。

此前 `codebase-retrieval` 对图谱摘要的支持还处于轻集成阶段：

- 默认关闭
- 调用方必须显式传 `include_graph_context: true`
- text / json 返回才会追加轻量直接图谱摘要

这次调整后：

- `include_graph_context` 默认改为开启
- MCP 注册元数据同步为默认开启
- 回归测试同步覆盖默认开启语义
- 轻量图谱摘要正式进入默认 `codebase-retrieval` 出口

这一步不是新增图谱能力，而是把刚刚落地的图谱能力从“独立工具”推进成“默认检索结果的一部分”。

### 4. 同步文档与外部说明入口

这部分由 `a5fc0f2` 和 `cdf1d42` 对应的文档改动共同完成。

重点包括：

- 恢复 README 中的交付文档链接
- 把 4 月 10 日更新写入 changelog
- 同步 MCP 文档和 README 索引

这一步的意义是避免出现“代码已变、对外说明还停留在旧状态”的漂移。

## 四、解决了什么问题

这次更新实际解决的是一组串联问题，而不是单一问题：

- **运行时基础不够稳**
  如果 cached 资源清理、限流锁、并发配置隔离和错误边界不稳，后续图谱接入主路径会放大复杂度。
- **检索失败时错误语义不够清晰**
  MCP 工具在 retrieval stage 失败时，调用方难以区分网络、配置和响应边界问题。
- **query-aware 检索在并发下存在互相污染风险**
  多请求并发时，搜索配置和运行时 wiring 需要明确隔离。
- **ContextAtlas 之前缺少内置代码图谱分析能力**
  在 4 月 10 日之前，系统还无法用统一内置管线完成 symbol extraction、impact analysis、change detection 和 execution tracing。
- **图谱能力与默认检索主路径割裂**
  即使图工具存在，最常见的 `codebase-retrieval` 默认结果仍然看不到图信号。
- **文档入口可能落后于实际交付状态**
  README / changelog / MCP 文档如果不同步，外部使用者会看到与系统行为不一致的说明。

## 五、边界、兼容性与当前状态

这次更新虽然覆盖面大，但仍然刻意保持了边界：

- 图谱进入默认检索主路径的方式，是“轻量直接图谱摘要”，不是完整图遍历结果
- `codebase-retrieval` 没有被扩成重型 graph query 工具
- 如果某些客户端更关注最小响应体，仍可显式传 `include_graph_context: false`
- 图谱能力虽然已经内置，但并不意味着 ContextAtlas 要立刻变成纯图数据库式产品

截至 2026-04-10 这轮收口后，更准确的状态表述是：

- 检索运行时链路更稳了
- 代码图谱分析管线已经进入仓库主代码
- 图谱能力第一次进入默认检索出口
- ContextAtlas 从“只有检索和记忆”推进到“检索 + 记忆 + 图谱分析”主路径
