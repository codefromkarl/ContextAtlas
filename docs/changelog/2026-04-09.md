# ContextAtlas 更新总结（2026-04-09）

本轮更新围绕两个核心收口目标推进：

> 让 `index:plan` / `index:update` 的升级策略更可解释、可调参；让长期记忆从 project-meta blob 升级为真正可查询、可治理的数据库能力。

## 一、为什么要更新

这次调整集中回应了几类已经出现的真实问题：

- 索引计划虽然能区分 full / incremental，但升级条件还不够显式，难解释也难调参
- 长期记忆仍停留在 project-meta blob 里，不利于检索、治理和迁移
- 默认路径与运行时目录创建存在冻结时机问题，容易在真实环境里踩到
- 运维和告警入口对同类健康数据的阈值口径还不完全一致
- dist 产物与依赖 dist 的回归测试之间存在偏移风险

## 二、这次更新的目标

这轮更新希望同时解决 5 个问题：

1. 让索引策略信号显式化、可解释、可调参
2. 让长期记忆升级为可查询、可治理的正式数据结构
3. 让默认路径和运行时初始化更稳
4. 让索引策略进入运维面板，而不是停留在单独命令里
5. 让发布门禁和告警阈值口径保持一致

## 三、如何更新

本轮主要通过 5 组动作推进：

### 1. 索引策略升级

- `analyzeIndexUpdatePlan()` 显式计算 `changedFiles`、`churnRatio`、`estimatedIncrementalBytes`、`estimatedFullBytes`、`incrementalCostRatio`、`fullRebuildTriggers`
- 当 churn、成本比和最小门槛满足时，会把计划升级为 `full`
- `strategySignals` 进入 `contextatlas index:plan --json`、`contextatlas index:update --json` 与文本计划报告

### 2. 长期记忆数据库化

- 把长期记忆从 `project_memory_meta` 里的 `global:<type>` blob 迁到 `long_term_memories` 与 `long_term_memories_fts`
- 保留读兼容
- 一旦发生写入，自动执行写迁移并清理 legacy blob
- 搜索策略升级为 FTS 优先、表内打分兜底

### 3. 运行时健壮性修复

- `MemoryHubDatabase` 改为在构造时动态解析默认 DB 路径并创建父目录
- benchmark / up-to-date 夹具改成显式种入 `vector_index_hash`、embedding dimensions 和 vector store presence

### 4. 阈值配置化与运维面接入

- 新增 `INDEX_UPDATE_CHURN_THRESHOLD`
- 新增 `INDEX_UPDATE_COST_RATIO_THRESHOLD`
- 新增 `INDEX_UPDATE_MIN_FILES`
- 新增 `INDEX_UPDATE_MIN_CHANGED_FILES`
- `health:check` 和 `ops:summary` 现在会直接展示策略摘要，包括建议模式、`changedFiles`、`churnRatio`、`incrementalCostRatio`、`fullRebuildTriggers`

### 5. 当日补充修复

- 重新把 `src/scanner/index.ts` 纳入 build / build:release / dev 入口，修复 dist 与 smoke / crawler 测试对齐问题
- `alert:eval` 新增 `--stale-days <days>`，并与 `ops:summary` 统一通过同一套告警指标构造逻辑

## 四、解决了什么问题

这次更新主要解决了以下问题：

- **full / incremental 升级依据不透明**
  运维和值班很难判断为什么某个仓库应该走 full。
- **长期记忆能存，但不好查、也不好管**
  blob 结构不利于检索、迁移和治理。
- **默认路径和运行时初始化不稳**
  真实环境里容易因为路径冻结时机不对而出现异常。
- **发布门禁与 dist 产物可能漂移**
  源码能过，不代表依赖 dist 的 smoke 与 crawler 测试一定能过。
- **运维摘要与告警文本可能给出不同结论**
  当 stale 阈值口径不一致时，同一仓库会出现两套判断。

## 五、边界、兼容性与当前状态

这次更新重点是基础设施解释性和治理能力，不是新增一套全新用户交互路径：

- 保持 `index:plan` / `index:update` 的核心入口不变
- 保持长期记忆对 legacy blob 的读兼容
- 重点增强的是可解释性、可调参能力、数据库化治理与运维一致性

截至本轮收口后，ContextAtlas 在“索引决策”和“长期记忆治理”两个层面都更像稳定基础设施，而不是一组难解释的功能集合。
