# ContextAtlas 更新总结（2026-04-07）

本轮更新围绕索引链路做了系统性收口：

> 让索引计划更轻、快照复制更省、失败恢复更清晰、队列更可观测、分片 fallback 更稳、存储更精简，并为后续优化建立可重复的性能基准。

## 一、为什么要更新

这次调整的背景集中在索引主路径的几个真实问题上：

- `index:plan` / `index:update` 的前置判断偏重，频繁调用成本高
- staging snapshot 存在明显复制放大
- 故障恢复虽然能做，但很难快速判断“谁能自动修、谁要人工修”
- 队列和 daemon 缺少专门的观察入口
- AST 失败、空 chunk 和 fallback 失败的语义混在一起，容易把解析失败误判成已收敛
- 存储冗余已经存在，但缺少量化入口
- 优化效果缺少稳定、可重复的基准验证方式

## 二、这次更新的目标

这轮更新希望同时解决 6 个问题：

1. 把索引计划阶段做轻，降低日常使用成本
2. 减少 staging snapshot 的磁盘放大
3. 把健康检查和任务观察面从“能修”推进到“看得见、知道怎么修”
4. 让 chunk fallback 与空内容语义更可靠
5. 先量化再裁剪存储冗余
6. 为后续优化建立可重复的 benchmark 基线

## 三、如何更新

本轮按 7 个阶段推进，核心动作如下：

### 1. 计划阶段轻量化

- `index:plan` / `index:update` 优先基于文件列表差异、`mtime`、`size`、`hash`、`vector_index_hash` 和 schema / snapshot 状态判断
- 只有确实需要时才进一步读取文件内容

### 2. staging snapshot 按需准备

- vector artifact 改为按需准备
- 保留 current / staging 原子切换语义
- 避免无意义复制 `index.db` / `vectors.lance`

### 3. 健康检查与任务观察面增强

- `contextatlas health:check` 把问题区分为 `Auto Fixable` 和 `Manual Checks`
- 新增 `contextatlas fts:rebuild-chunks --project-id <id>`
- 新增 `contextatlas task:status` 和 `contextatlas task:inspect <taskId>`
- `contextatlas start` 在索引未完成时直接给出队列和排障入口

### 4. 空 chunk 与 fallback 语义收口

- 把“真实空内容”和“AST 失败 / splitter 返回空 / fallback 失败”明确拆开
- 只有真实空内容才允许直接标记为已收敛
- 扩大 fallback split 适用语言范围，减少多语言仓库中的检索盲区

### 5. 存储冗余量化与裁剪

- 新增 `contextatlas storage:analyze`
- 量化 `files.content`、`files_fts.content`、`chunks_fts.content`、LanceDB `display_code` 和 `vector_text`
- 先裁掉一刀低风险冗余：新写入记录不再持久化 `vector_text`

### 6. 离线 benchmark 基线建立

- 新增 `contextatlas perf:benchmark`
- 支持单场景和 `--matrix` 全矩阵模式
- release smoke 增加 `benchmark-small-noop`

## 四、解决了什么问题

这次更新主要解决了以下问题：

- **索引计划太重**
  计划阶段经常接近正式扫描，导致频繁使用时成本偏高。
- **快照复制放大**
  大仓库索引时会产生不必要的磁盘开销。
- **故障恢复不透明**
  出问题时很难第一时间知道该自动修还是人工修。
- **解析失败被静默吞掉**
  parse failure 容易被误标为已收敛，后续看起来像“系统没问题，结果却缺失”。
- **存储优化缺少依据**
  没有量化入口，就很难判断哪些冗余是可安全裁剪的。
- **优化结果难以回归**
  没有稳定 benchmark，就无法把“感觉更快”转成“能验证更快”。

## 五、边界、兼容性与当前状态

这次更新聚焦的是索引主路径收口，不是重写整套索引架构：

- 保持现有 current / staging 切换语义
- 保持 `index:update` 的执行含义不变
- 重点改进计划判断、恢复可见性、fallback 语义和观测能力

截至本轮完成后，索引系统已经从“能工作”推进到“更轻、更透明、更容易修复，也更容易量化”。

## 六、涉及文件、命令与入口

本轮涉及的主要入口包括：

- `contextatlas index:plan /path/to/repo --json`
- `contextatlas index:update /path/to/repo`
- `contextatlas health:check`
- `contextatlas task:status`
- `contextatlas task:inspect <taskId>`
- `contextatlas fts:rebuild-chunks --project-id <id>`
- `contextatlas storage:analyze --project-id <id>`
- `contextatlas perf:benchmark --size small --scenario noop --json`
- `contextatlas perf:benchmark --matrix`
