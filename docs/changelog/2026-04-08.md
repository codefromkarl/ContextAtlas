# ContextAtlas 更新总结（2026-04-08）

本轮更新围绕两个相关方向收口：

> 先把 embedding gateway 从“最小可用转发层”推进到“稳定本地接入层”，再把上下文生命周期从“有能力点”推进到“有明确唤醒协议”。

## 一、为什么要更新

这次调整背后有两条主线：

- embedding 访问层虽然已经能转发，但在 failover、缓存、并发去重和可观测性上还不够稳定
- ContextAtlas 自身的记忆、证据、checkpoint、handoff 和 diary 能力虽然逐步增多，但还缺少统一的上下文唤醒结构

因此这次不是只补一个 gateway，而是同时收口“模型上游接入层”和“内部上下文生命周期”。

## 二、这次更新的目标

这轮更新希望同时解决 4 个问题：

1. 让 embedding gateway 可以作为稳定的本地接入层使用
2. 让相同 embedding 请求支持缓存复用与并发去重
3. 让 provider 和 cache 状态进入可观测路径
4. 让 ContextAtlas 的上下文装配、证据、时态事实和 diary 进入统一生命周期

## 三、如何更新

本轮主要通过两组动作推进：

### 1. embedding gateway 能力收口

- 新增 `contextatlas gateway:embeddings` CLI 入口
- 提供 OpenAI-compatible `POST /v1/embeddings`
- 支持多上游配置、加权轮询、网络错误 / `429` / `5xx` 自动 failover
- provider cooldown 可临时摘除异常上游
- 支持可选 Bearer Token 鉴权
- `GET /healthz` 暴露 provider 与 cache 健康状态

缓存与请求复用层面：

- 把缓存抽象成统一 cache manager
- 支持 memory、Redis、hybrid 三种 backend
- 保留 inflight request coalescing，避免并发相同请求重复打上游
- 响应头区分 `miss` / `hit` / `coalesced`

### 2. 上下文生命周期主路径收口

- `assemble_context` 正式输出 `L0/L1/L2/L3` wakeup layers
- `temporal-fact / journal / evidence` 进入长期记忆正式模型
- 新增 `record_agent_diary / read_agent_diary / find_agent_diary`
- retrieval 主路径显式展示 `evidence` 与 `temporal-fact`
- `prepare_handoff` 可携带 `supportingRefs` 与按需 diary 摘要
- feature / decision / long-term memory 写入前新增重复提示与冲突诊断

## 四、解决了什么问题

这次更新主要解决了以下问题：

- **embedding 上游不稳定时缺少本地缓冲层**
  之前 gateway 更像透传层，provider 失败会更直接地暴露给调用方。
- **相同请求会重复放大到上游**
  没有稳定的缓存和并发合并时，吞吐和成本都会受影响。
- **cache 与 provider 状态不可见**
  即使 gateway 能用，也难以快速判断命中情况、inflight 数量和 provider 健康度。
- **上下文生命周期能力分散**
  证据、时态事实、checkpoint、handoff、diary 虽然存在，但还不是统一主路径的一部分。

## 五、边界、兼容性与当前状态

这次更新刻意保持了两个边界：

- gateway 仍然是 ContextAtlas 的本地 embeddings 接入层，而不是独立的 provider orchestration 平台
- 上下文生命周期重点是“分层唤醒”和“证据进入主链路”，不是引入新的任务编排系统

截至本轮结束，ContextAtlas 已经在两条链路上形成更稳定的主路径：

- embeddings 侧：多上游容灾 + cache + coalescing + health
- context 侧：wakeup layers + evidence + temporal facts + diary + handoff

## 六、涉及文件、命令与入口

本轮涉及的主要入口包括：

- `contextatlas gateway:embeddings --port 8787 --cache-backend redis --redis-url redis://127.0.0.1:6379/0`
- `GET /v1/embeddings`
- `GET /healthz`
- `assemble_context`
- `prepare_handoff`
- `record_agent_diary`
- `read_agent_diary`
- `find_agent_diary`
