# ContextAtlas 更新总结（2026-04-09）

本轮更新围绕两个核心目标推进：

> 让 `index:plan` / `index:update` 的升级策略更可解释、可调参；让长期记忆从 project-meta blob 升级为真正可查询、可治理的数据库能力。

---

## 一、本轮完成了什么

本轮主要完成了 5 个方向的收口：

1. **索引更新计划新增 churn / cost 策略信号**
2. **长期记忆迁移到独立表 + FTS5**
3. **默认路径与运行时目录创建逻辑硬化**
4. **索引阈值配置化**
5. **文档、模板与部署入口同步**

---

## 二、索引策略升级

`analyzeIndexUpdatePlan()` 现在不再只看“有没有文件变化”，还会显式计算：

- `changedFiles`
- `churnRatio`
- `estimatedIncrementalBytes`
- `estimatedFullBytes`
- `incrementalCostRatio`
- `fullRebuildTriggers`

当以下条件满足时，计划会升级为 `full`：

- churn 比例超过阈值
- 估算增量成本接近全量成本
- 且仓库规模、改动规模都达到最小门槛

这部分现在也会通过 `strategySignals` 暴露给：

- `contextatlas index:plan --json`
- `contextatlas index:update --json`
- 文本版计划报告

---

## 三、长期记忆数据库化

长期记忆原先存放在 `project_memory_meta` 里的 `global:<type>` JSON blob。

现在已经迁到独立结构：

- `long_term_memories`
- `long_term_memories_fts`

同时保留两层兼容：

- **读兼容**：旧 blob 仍可读取
- **写迁移**：一旦发生写入，会把 legacy 项迁到新表并清理旧 blob

搜索策略也升级为：

- FTS 命中优先
- 表内内存打分兜底

---

## 四、运行时健壮性修复

本轮顺手修了两类真实环境里会踩到的问题：

### 1. Memory hub 默认路径不再在模块加载时冻结

`MemoryHubDatabase` 现在会在构造时动态解析默认 DB 路径，并自动创建父目录。

### 2. 索引 benchmark / up-to-date 夹具更稳定

相关测试夹具改成显式种入：

- `vector_index_hash = hash`
- embedding dimensions
- vector store presence

这样测试验证的是明确状态，而不是环境偶然性。

---

## 五、阈值配置化

新的索引策略参数已经从硬编码外置成环境变量：

```bash
INDEX_UPDATE_CHURN_THRESHOLD=0.35
INDEX_UPDATE_COST_RATIO_THRESHOLD=0.65
INDEX_UPDATE_MIN_FILES=8
INDEX_UPDATE_MIN_CHANGED_FILES=5
```

当前这些参数已经同步到了：

- `contextatlas init` 生成的默认 `.env`
- `README.md`
- `README.EN.md`
- `docs/reference/cli.md`
- `docs/guides/first-use.md`
- `docs/guides/deployment.md`

## 六、运维面继续推进

在主交付链完成后，本日又补了一步收口，把索引策略信号正式接入到了运维观察面：

- `health:check`
- `ops:summary`

具体来说：

- `analyzeIndexHealth()` 现在会从索引队列里回溯每个项目最近一次已知的 `repo_path`
- 基于该路径实时调用 `analyzeIndexUpdatePlan()`
- 把当前策略摘要挂到每个项目的 snapshot health 上
- 在文本报告里直接展示：
  - 当前建议模式：`none / incremental / full`
  - `changedFiles`
  - `churnRatio`
  - `incrementalCostRatio`
  - `fullRebuildTriggers`

这一步的意义不是再增加一个独立入口，而是让团队在日常运维视角里就能看到：

- 为什么某个仓库应该走 `full`
- 为什么某个仓库仍适合 `incremental`
- 当前阈值信号是否已经逼近升级边界

对应改动主要落在：

- `src/monitoring/indexHealth.ts`
- `src/monitoring/opsSummary.ts`
- `tests/index-strategy-ops.test.ts`

## 九、对团队的实际意义

这轮更新之后，ContextAtlas 在“索引决策”和“长期记忆治理”两个层面都更像基础设施，而不是一组能跑但难解释的功能集合：

- **索引策略可解释**
- **索引策略可调参**
- **索引策略进入运维面板**
- **长期记忆可检索**
- **长期记忆可迁移**
- **默认路径更稳**
- **部署入口更完整**

如果下一步要继续演进，优先建议：

1. 把 `index:diagnose` 接入 README / first-use / runbook 等主路径文档，减少新用户只看 `index:plan` 才能发现阈值配置的情况
2. 继续把治理联动扩到更多入口，保证 `alert:eval`、`ops:summary`、`health:full` 对 memory health 的口径长期一致
3. 在现有 catalog drift + orphaned memory + stale memory 场景化 smoke 夹具基础上，继续补更多组合场景，例如 shared/profile governance 差异或 mixed-signal 告警，验证摘要与告警文本的优先级和组合表现
